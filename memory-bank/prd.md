# 需求文档

## 项目背景

LarkMemory 是飞书 AI 比赛 OpenClaw 赛道下的项目，课题是“企业级长程协作 Memory 系统”。

赛道定位是在飞书里“养虾”，让 AI 拥有长时记忆与主动服务能力，打造专属办公 Agent 伙伴。课题聚焦企业跨部门长程协作中的智能体“失忆”、信息断层和协作断层问题，要求基于飞书 OpenClaw、飞书 CLI、飞书生态与大模型能力，打造具备全流程记忆管理的企业级记忆引擎。

## 课题目标

项目需要完成三类核心能力：

- 定义记忆：说明企业环境下什么是有价值的个人/团队记忆。（是 CLI 的⾼频命令？是⻜书⽂档⾥的架构设计？是群聊⾥的审批习惯？还是⽤⼾的隐式偏好？）
- 构建引擎：实现记忆的提取、存储、检索、更新和遗忘。（如何保证这套系统在 CLI 和⻜书中⽆缝流转？）
- 证明价值：设计评测用例和指标，证明系统真的“记住了”，并产生实际效能。（你需要⾃⼰设计⼀套评测⽤例和指标，来跑通你的逻辑。）

## 当前方向

当前项目优先围绕企业级长程协作 Memory 系统构建通用 Memory Engine。实现时以项目决策与上下文记忆作为重点 demo 场景之一，同时保留 CLI 工作流、个人偏好、团队长期保留记忆等 domain 的扩展空间。

#### ⽅向 A：CLI ⾼频命令与⼯作流记忆（偏开发者效率）

- 场景描述：开发者在终端中频繁输⼊⻓命令、复杂参数或项⽬路径。设计记忆系统，⾃动记录⽤⼾的⾼频命令模式、常⽤参数组合、项⽬路径偏好，并在⽤⼾输⼊前缀时⾃动补全或推荐。更进⼀步，能记住“在项⽬ A 中部署使⽤参数 X，在项⽬ B中使⽤参数 Y”，实现上下⽂感知的命令建议。 
- 记忆特征：显式记忆（⽤⼾可主动教）+ 隐式记忆（系统统计频率⾃动总结）

#### ⽅向 B：⻜书项⽬决策与上下⽂记忆（偏项⽬管理）

- 场景描述：在⻜书项⽬群聊或⽂档讨论中，经常出现“之前我们决定⽤⽅案 B ⽽不是 A”、“上周五确认了截⽌⽇期是 5号”等决策性信息。设计记忆系统，从对话/⽂档中⾃动提取决策、理由、反对意⻅、结论，并建⽴与项⽬阶段、时间点的关联。当后续讨论触及相关话题时，主动推送“历史决策卡⽚”，避免重复争论或遗忘。
- 记忆特征：结构化记忆（决策-理由-结论）+ 时序关联 + 主动检索与推送

#### ⽅向 C：个⼈⼯作习惯与偏好记忆（偏个性化体验）

- 场景描述：⽤⼾在⻜书中有固定的⼯作习惯，例如“每周五下午⾃动整理周报”、“开会前 5 分钟提醒我准备材料”、“优先使⽤表格视图⽽⾮列表”。设计记忆系统，通过观察⽤⼾⾏为（如点击、命令、⽇程安排），⾃动学习其偏好和规律，并在适当时机主动提供个性化建议或⾃动化执⾏。
- 记忆特征：隐式学习 + 规则⽣成 + 主动服务

#### ⽅向 D：团队知识断层与遗忘预警（偏组织协同）

- 场景描述：团队项⽬中，某些关键信息（如 API 密钥更新、客⼾偏好、竞品动态）⼀旦被提及，若没有有效记忆，⼏周后会被全员遗忘，导致错误。设计记忆系统，允许团队“注⼊”需要⻓期记住的事项，并设置遗忘曲线（如 Ebbinghaus 模型），当某事项即将被遗忘时，在群聊中主动进⾏“复习提醒”。同时⽀持“记忆覆盖”（如信息更新后⾃动废弃旧版本）。
- 记忆特征：团队共享记忆 + 遗忘曲线 + 版本管理

## 用户价值

- 减少跨会话、跨团队协作中的上下文断层。
- 避免重复讨论已经确定的项目决策。
- 帮助 Agent 在后续任务中引用正确历史背景。
- 支持长期记忆的更新、覆盖和遗忘，避免误用过期信息。
- 用评测数据证明记忆系统带来的效率提升。

## 核心功能

- 标准化事件接入：接收来自 OpenClaw、CLI、飞书聊天、飞书文档、会议纪要等来源的事件。
- 记忆抽取：从事件中提取有长期价值的结构化记忆。
- 记忆存储：保存统一 MemoryCore 和 domain-specific payload。
- 记忆检索：根据 query、项目、用户、时间、主题和上下文召回相关记忆。
- 记忆更新：处理重复记忆、相似记忆、冲突记忆和时序覆盖。
- 记忆遗忘：支持候选、活跃、被覆盖、过期、遗忘等状态。
- 主动服务：在后续相关讨论或任务中主动推送相关记忆卡片。
- 评测验证：支持抗干扰测试、矛盾更新测试和效能指标验证。
